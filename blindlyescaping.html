Blindly-escaping is insufficient

JavaScript Templating blindly escapes every output using a single filter for XSS mitigation


What's Blindly-escaping?

Template engines such as Handlebars and DustJS mitigate Cross-Site Scripting (XSS) by escaping a fixed set of sensitive characters from potentially untrusted inputs, and thus preventing browsers from interpreting them as HTML entities, which otherwise could lead to malicious script executions.

While the approach can mitigate some XSS vulnerabilities, it may give a false sense of security that the XSS problem is completely solved.

What's wrong?

More characters could become sensitive depending on the output contexts. For instance, a space character is XSS-sensitive in an unquoted attribute value context.

Example of Vulnerability

  • Visit the Handlebars demo page
  • Use <input value=undefined/> as template
  • Use {"untrusted": "x onblur=alert(1)"} as data input
  • Compile. The output is <input value=x onblur=alert(1)/>
  • When run at browsers, expect an alert when the input loses focus.

The table below summarizes a few attack vectors to bypass the blindly-escaping filter.

Output ContextSensitive Characters
In Unquoted Attribute ValuesSpace, Null, Empty, etc
In URI contexts (e.g., <a href="{{url}}">)javascript: URI
In style attribute (e.g., <div style="background:{{value}}">)url(javascript: URI)
etc...

None of these characters are escaped by the blindly-escaping filter. Nevertheless, simply including all of them for escaping in one filter regardless of the output contexts is overkill. For instance, the space character is arguably a legitimate user input, but could also become a sensitive character depending on the output context.
